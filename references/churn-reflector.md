# churn_reflector · 静默流失根因归因模块

> 关④（churn-root-cause-review.md，2026-07-11 已签批）。配套 `skill-forge` 的叠加能力模块：
> 当用户**无声离开**某技能时，用行为代理特征推断"为什么走"，产出**可解释假设 + 证据链**，
> 反哺技能迭代，闭合增长飞轮出口。
>
> **铁律**：本模块是**叠加能力**，不改动锻造内核、不 bump 版本、不改 changelog。
> 与关②（萃取器，默认**关闭**）**机制同族但 opt-in 默认值相反**——关④ 默认**开启**。

---

## 〇、为什么存在

- 现有闭环只收"正向/显式"信号（confusion/helpful/unhelpful/suggestion/misdiagnosis），
  缺"用户无声离开"这半环。流失用户带着不满静默消失，技能永远学不到自己哪里不行。
- 关④ 是关③（进迭代循环）的**对称补半环**：关③ 收集"在车上"的信号，关④ 收集"下车"的信号。
  两者汇入同一条 `signal → distill → proposal → 用户审 → 重发布` 闭环。
- **核心洞察（用户原话）**：用户无声的离开也是一种选择，但我们要反思如何迭代，要把这个闭环跑通。

---

## 一、监控边界（用户拍板：默认开，但有红线）

| 项 | 规格 |
|----|------|
| **opt-in 默认** | **开启**（与关② 默认关闭相反，已显式标注差异） |
| **数据范围** | 仅 method-layer 元数据（调用/终态/卡点/版本），**零 PII、不读对话原文** |
| **首次告知** | 首次启动一句透明说明（见 §六） |
| **可关闭** | 用户可要求"别做流失分析"，立即永久静音本模块 |
| **云上传** | 仅 `cloud` 模式才上云；`local` 模式全留本机、零外泄 |
| **匿名聚合** | cloud 模式跨用户聚合仅用 reason + method_layer（零 PII） |

---

## 二、如何判定"停止"（冷却期 + 历史深度）

满足**全部**才进入归因（避免误伤活跃用户）：
- **冷却期**：该 `skill_slug` 距最后一次调用 ≥ **14** 天（`CHURN_COOLDOWN_DAYS`）。
- **历史深度**：曾调用 ≥ **2** 次（`CHURN_MIN_CALLS`），排除"误装即走"噪声。
- **曾达终态**：至少 1 次 `completed=True`（说明技能可用，流失非纯 bug）。

若不满足 → 不做任何事（保持沉默，优于打扰）。

---

## 三、归因分类法（7 类 + E 前置过滤）

规则 + 可解释打分（**不用神经网络**，小数据下黑盒不可信也不可审）。每条结论带证据链。

### E 类 · 自然完成（前置过滤，保守）
- **判定**：调用 ≤ 2 次（`CHURN_MIN_CALLS_REUSE`）且无相似场景复现 → 视为一次性任务，直接排除，**不迭代、不发信号**。
- **为什么重要**：E 是最大头，必须先排除，否则把"做完"误当"失败"去瞎改——这正是假阳性最贵的情形。

### C1 · 找不到触发时机（Discovery Gap）
- 技能有用，但同类需求出现时用户没调用它（冷却期内出现过相似意图词却没触发）。
- 迭代方向：frontmatter 加 `use_when`；相关上下文主动轻提醒。

### C2 · 卡在某一步（Friction / Stuck Step）
- 调用多次未达终态，`stuck_step` 集中在同一步骤（如"3 次 2 次停在第 2 步"）。
- 迭代方向：该步骤加示例 / 前置检查 / 拆步 / 失败兜底。

### C3 · 结果不合预期（Output Mismatch）
- 跑通（completed=True）但后续同类任务用户改用别的方式（转人工/手写），或同窗口内不再复用。
- 迭代方向：调输出约束、加偏好默认值、补格式/风格示例。

### C4 · 信任断裂（Trust Break）
- session 内曾出现 `confusion`/`misdiagnosis` 信号，之后该技能调用归零。
- 迭代方向：加校验 / 边界说明 / 免责提示 / 危险动作二次确认。

### C5 · 被替代（Substituted）
- 弱证据：用户改用另一技能/手动/别的产品（需跨技能遥测或"问用户"确认）。证据弱 → 低于阈值转"问用户"。
- 迭代方向：差异化卖点 / 补齐短板。

### C6 · 自身回归（Self-Regression，G 类自检）
- `skill_version` 戳 + 发版后完成率/复用率骤降（> 30%，`REGRESSION_DROP`）。
- 迭代方向：**回滚该版本** + 质量门加固。最伤口碑、也最该被自动捕获。

### C7 · 未知 / 证据不足（Unknown）
- 不满足以上任何类明显特征，或冷却期刚到、历史过短 → 不自动迭代，转"问用户"或沉默。

## 四、置信度打分（透明、可复算）

每个归因给一个 `confidence`（0–1），**必须附证据链**。参考基线：

| 类别 | 触发证据 | 建议 confidence |
|------|----------|-----------------|
| C2 | ≥2/3 次停在同一步 | 0.7 |
| C4 | 有 confusion/misdiagnosis 后调用归零 | 0.7 |
| C6 | 发版后完成率降幅 >30% | 0.85 |
| C1 | 冷却期内相似意图未触发 | 0.65 |
| C3 | 跑通但同窗口不再复用 | 0.65 |
| C5 | 仅弱信号（改用别处） | 0.5 |
| C7 | 证据不足 | 0.4 |

门限：`CHURN_CONFIDENCE_GATE = 0.6`（用户拍板）。
- **≥ 0.6** → 自动封装 `event=abandoned` 信号，进蒸馏 → 提案。
- **< 0.6** → **不自动迭代**，转"问用户"卡片（§五）。

> 代价结构：**假阳性（瞎改）比假阴性（漏判）更贵**。E 类前置保守过滤 + 0.6 阈值 + 问用户分支，三层防瞎改。

---

## 五、"问用户"卡片（< 0.6 时）

```
🤔 你最近 16 天没再用「<技能名>」了。
我们猜：是不是上次卡在第 2 步（<卡点描述>）？
［对，就是这原因］  ［不是，我其实…］  ［别再问这类］
```
- 用户回"对" → 采纳假设、补强证据、立即进提案。
- 用户回"不是，我其实…（忘了/换工具/做完了）" → 记入提案 `user_note`（打回意见回填），**反哺分类器**（元蒸馏，越判越准）。
- "别再问" → 该技能永久静音此分支。

**频控**：每技能冷却期内最多 1 次问询，避免骚扰。

## 六、本地遥测约定（.usage_cache.json）

仅 method-layer 元数据，可随时删。落在技能可写数据区（如 `~/.workbuddy/skills/<slug>/.usage_cache.json` 或 Forge 工作区）：

```json
{
  "<skill_slug>": {
    "call_ts": ["2026-06-20", "2026-06-25", "2026-07-01"],
    "completed": [true, false, false],
    "stuck_step": 2,
    "current_version": "2.3.0"
  }
}
```
- 技能被调用时增量更新 `call_ts` / `completed` / `stuck_step` / `current_version`。
- churn_reflector 巡检（cloud=distill-cron 定时；local=用户返回技能或主动"审计我的技能"时）读取它做归因。
- **不读对话原文、不存 PII**；纯 method-layer。

---

## 七、证据链格式（每条结论必须附）

```json
{
  "reason": "C2",
  "confidence": 0.72,
  "evidence": [
    {"type": "cooldown", "text": "距末次调用 16 天（阈值 14）", "ts": "2026-07-11"},
    {"type": "stuck",   "text": "近 3 次调用 2 次停在第 2 步", "ts": "2026-06-25"}
  ]
}
```

## 八、如何发出 abandoned 信号 / 提案

- **cloud 模式（零密钥，无需 token）**：经本技能 `cloud_config.json.ingest_url` + `POST {ingest_url}/ingest/anon` 匿名回传（`mode:"cloud"`，服务端按 slug 派生 `anon_id`），body：
  ```json
  {
    "slug": "<skill_slug>",
    "method_layer": "<卡点层，如 step.2>",
    "event": "abandoned",
    "weight": 1,
    "note": "<§七 JSON 序列化字符串>",
    "skill_version": "<current_version>",
    "anon_id": "<可省略，服务端按 slug 派生>",
    "mode": "cloud"
  }
  ```
  后端 `cjg-signal-ingest` 入库 → `cjg-distill-cron` 匹配 `rules.yaml` 的 `abandoned` 规则组 →
  写 `proposals(pending, 带 abandon_reason + evidence_chain)` → `cjg-proposal-notify` 通知用户本人 →
  用户在应用内审（先看到"我们为什么这么猜"再决定）→ 通过则 Forge 应用 + 质量门 → 重发布。
- **local 模式**（最保守、零外泄）：
  不联网。直接在对话内把"假设 + 证据链"作为提案呈现给用户审；或本地跑 `cjg-evo/backend/lib/distill.py`
  得到带 evidence_chain 的 proposal dict，再呈现。无 `user_id`、无云上传、无跨用户聚合。

---

## 九、质量门（防 churn 误判导致越炼越差劲）

- 关④ 提案同样过质量门（相关性 × 质量 × 加入后效果预评估）。
- C6 回归类：质量门额外校验"回滚版本确实存在且稳定"，避免回滚到更差的旧版。
- 证据不足（C7）或置信 < 0.6 的，**不进提案**——沉默优于噪声。

---

## 十、M0 范围与 M1+ 路线

- **M0（本次）**：单技能闭环（signal → distill → proposal 带 evidence_chain → 用户审）。
- **M1+**：① 用户打回反哺分类器（元蒸馏校准阈值/权重）；② 跨技能同因 → 改 Forge 模板本身（生产线自我进化）；③ 匿名 churn 模式进 `evolution_pool` 供订阅参考。
