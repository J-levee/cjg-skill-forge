# 真机锻造子模式 · 详细流程模板（Real-Machine Forge）

配套 `cjg-skill-forge` SKILL.md 的「子模式 A.5」。本文件是可复用的执行清单：每锻造一个技能，照此走 S0→S6。

> **🔴 定位铁律（最重要，先读）**
> - **目标永远是「全球最牛（Global-Best）」，不是比赛第一。** 比赛 / 平台 / 任何局部场景，只是**验证场（test vehicle）**——用来真实测试「这个技能能不能跑通」「锻造炉本身好不好用」。
> - **S3 外部标杆的范围 = 全世界**（互联网上所有技能/项目/工具/论文/知识）。**绝不可把标杆缩到某场比赛的参赛作品或某个平台的技能列表**——那是局部子集，会制造「我天下第一」的虚假安全感。

## 一、立项（S0 之前，先做减法）
- 明确真实场景（如某大赛主题、某生产部署需求、某用户痛点）。**把它当测试场，不当目标。**
- 写清该技能要解决的真实问题 + 全球差异化一句话，焊进 description 首屏。
- 确认依赖底座技能已安装且可真机运行（如 `global-biblio-base` 的网关密钥、`tmeet-skill` 的 CLI）。

## 二、S0 脚手架
- `skill-creator` 的 `init_skill.py` 起骨架。
- 强制纪律 11 三件套：① `§零` 进化燃料（安装即开）② footer（来源）③ `references/coverage.md`（覆盖维度表）。

## 三、S1 真实接线（不可跳过）
- 装好依赖技能（如 `tmeet-skill`），读其真实命令面，把**确切调用语法**写进 SKILL.md（命令名、参数、子命令）。
- 数据侧写清真实接口（如 `global-biblio-base` 用 `api_path` 全代理格式，非语义端点）。
- 设计层合规不够：必须写"调哪个能力 + 怎么调 + 调失败怎么办"。

## 四、S2 真机取证（不可跳过）
- 用真实账号跑通 1 条主链路。例：注册邮箱 → `/consume` → `/search` → 取真实返回。
- 把**真实返回**原样存入 `references/<skill>_evidence.md`：真实字段名、真实 Identifier、真实 prior work。
- 发现的接口坑（端点偶发空返回、字段名不符）立刻写回 SKILL.md 容错段落。

## 五、S3 外部标杆（**全球范围，不可缩到局部场景**）
- **先全网扫描**：用 WebSearch/WebFetch 扒同领域**全球**顶级公开作品——含学术论文、商业产品、开源项目、其他平台的技能。至少 ≥3 个真实竞品。
- 做对标表：场景 / 核心方法 / 数据规模 / 独有能力 / 我们的差距。
- 证差异化成立，写入 `references/benchmark.md`（标杆是全世界，不是比赛内部）。
- ⚠️ 自查：如果你的标杆表里只有"某场比赛的 N 个作品"，说明你犯了最常见的错误——退回重做。

## 六、S4 覆盖审计
- 用真实 ID（文献 `Identifier`、全球竞品名）核对 `coverage.md` 各维度。
- 确认无盲区；竞品扎堆的维度持平即可，独有维度（如我们的"会议实时"）形成壁垒，差距维度列入迭代路线图。

## 七、S5 生产签批
- 写评审文档（做什么/改哪些文件/真实来源/验收标准）→ 用户明确签批（纪律 5）。
- 触活系统（真实 API 写操作）的技能强制适用。

## 八、S6 校验打包
- `quick_validate.py` + `package_skill.py` 通过。
- 准备演示（模拟输入→真实输出）GIF/截图，应对"最短时间理解并跑起"。

## 九、通用真机策略（S2 经验 · 仅留通用方法论，smartlib 专属接口知识已移至 global-biblio-base）
1. **编排类技能的 S2 真机策略（来自 paper-club-pilot v2.0.0，2026-07-18）**：当技能是「编排层」——复用其他技能的能力（如 global-biblio-base 检索、腾讯会议智能纪要）做串联而非自建独立 API 时——S2 只需验证**最难的外部依赖**即可。paper-club-pilot 的会前推荐管线验证了 SmartLib 网关（/consume→/search/global 命中真实论文 2283 篇可推荐，配额 350→349）；而「会中捕获」依赖用户授权会议数据、「会后沉淀」是编排逻辑（非独立外部 API），二者标注为"非独立依赖"即可，无需强行跑通完整三阶段闭环。**教训：S2 验证的是"这个技能声称的核心能力是否真有外部支撑"，不是"整个工作流每一步都能演示"。**
2. **答辩/演练类技能的 S2 真机策略与全球竞品格局（来自 defense-rehearsal v2.0.0，2026-07-18）**：defense-rehearsal 的核心差异化是「文献短板扫描」（扫用户参考文献，标出「引用但不熟」的）——这是 VivaCoach / Verble / Sidangin / Yoodli 等全球竞品**无一具备**的能力（它们只基于用户上传稿做内部提问，不接真实文献库）。S2 验证了该链路：SmartLib 网关命中真实论文（图神经网络+交通预测方向 160 篇可检索，首篇 Identifier 2031308272125），配额真实计次（349→348）。腾讯会议录制侧标注为「需用户授权」，不强行走真机（编排逻辑）。**经验：答辩类技能的壁垒不在"模拟提问"（竞品都会），而在"用真实文献库做短板扫描"——这是本技能独有的杀手锏。**

## 十、评分维度映射（仅作"测试场"参考，非锻造目标）
> ⚠️ 本表只是"若用某场比赛当验证场，怎么拿分"的参考。**锻造目标永远是 Global-Best，不是比赛第一**。描述面向真实用户场景，**不写参赛/大赛刻意话术**（见 SKILL.md A.5 硬规则第 4 条）。

| 评审维 | 怎么拿 | 本流程对应 |
|--------|--------|------------|
| 场景契合度 | 描述面向真实使用场景（非参赛话术） | S0 立项 |
| 实用性 | 真机跑通 + demo | S1/S2/S6 |
| 创新性 | 全球独有壁垒 | S2/S3 |
| 完成度 | 全链路闭环 + 三件套 + 发布包安全审查 | S0/S4/S6/纪律13 |
| 生态联动 | 调官方 Skill（如顺带） | S1 |

> 注意：真正的毕业标准见纪律 2 + 评分尺的 Global-Best 闸门（≥85 且同时具备全球标杆 + 真机测试）。比赛分数只是附加证据。

## 十一、多源融合连贯性闸门（纪律 12，v2.7.4 新增）
> 当技能**融合多个来源**（其他技能 / 别人设计 / 外部想法）时，在 S4 覆盖审计同步跑本闸，防止"缝合怪"与"假设冲突静默拼合"。

**三道强制判据（每 fused piece 在 coverage.md 追加三字段）：**
1. **来源假设对齐**：抽每个融合源的内隐假设（用户层级 / 输入输出契约 / 失败处理 / 默认世界观）。发现冲突先裁决再继续，禁止带冲突进 S5。
2. **边际收益门槛**：每新增源必须答「填了 coverage.md 哪个维度 C + 真机增益可观测？」，答不上来不 fuse。
3. **反缝合怪**：融合后若出现「谁都不负责」的孤儿中间态 → 判 incoherent，退回 S0/S1 重熔，不得进 S5 签批。

**诚实边界**：本闸拦得住缝合怪与假设冲突，但**拦不住「分类法本身列错」**——那需领域专家（用户）在 S3/S3 把关；本闸是兜底，不替代人的判断。

**实例（novelty-validator v1.5）**：融合 tmeet（会议转写）+ global-biblio-base（12亿文献）+ sweeping-monk（创新五轴）。三源假设对齐：tmeet 默认"用户是会议参与者"、biblio 默认"用户要查文献"、monk 默认"用户是科研决策者"——三者层级一致（均为科研用户），无冲突；边际收益清晰（tmeet→C1输入源 / biblio→C4数据源 / monk→C15创新路径）；无孤儿中间态 → 通过。

## 十二、发布包安全审查（见纪律 13）
每锻造一个技能发布前，强制过一遍纪律 13 清单：剔除 `config.json` / 含密钥文件 / `__pycache__` / `.backup` / PII（邮箱/账号/姓名）/ 锻造内部文档（`*_evidence.md` / `benchmark.md` / `coverage.md` / `engineering-notes.md`）→ 修复 SKILL.md 中指向已删文件的死链 → 合规检查（图标无品牌无文字、≤5MB）。novelty-validator 首版发布即因漏做此步，把用户邮箱 `user@example.com` 随包外泄、且带了 API key 文件风险——现已固化为强制纪律。
