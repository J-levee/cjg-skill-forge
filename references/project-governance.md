# project-governance.md — SkillForge 通用项目治理参考

> 本文件是 SkillForge 的子参考，把 `project-dev-constitution.md`（藏经阁·易筋平台开发宪法）中的**通用部分**抽出，供 Forge 任何技能时遵守。
> 剔除了 CJG 平台专属内容（tenant_id / M0-M4 映射 / 脱敏上行后端等，那些属于平台项目，不属于通用生产线）。
>
> **来源（三流合一）**：
> - **A. vibe-coding 治理**（抖音博主「银二码」sliver-vibe-coding 技能，Apache-2.0）
> - **B. SmartLib 实战**（二次签批铁律、执行层确定性、逃生舱、分层架构）
> - **C. skill-forge 元方法**（拟真测试、非侵入反馈环、覆盖审计零编造、度量尺自审）
>
> SkillForge 本身也受本文件约束（生产线守宪法，造出的技能才天生合规）。

---

## 一、总则（不可违背的四条底线）

1. **立项优先，禁止裸奔写码**（A+B）
   任何功能开发前，必须有：目标用户 / MVP 定义 / 功能边界 / **"不做什么"清单**。缺一项不得直接写实现。

2. **人定方向，AI 定落点**（B）
   业务结果与重大取舍由用户确认；技术落点、验证方法、细节实现由 AI 负责。**不确定先问，不猜**。

3. **生产改动先评审后签批**（B，最硬）
   任何生产技能文件 / 配置改动，先有评审文档 + 用户签批才动手。用户说"继续"= 按既定方案继续不扩范围；"停止"= 立即停手并主动回滚。

4. **证据零编造**（C）
   取号、引用、验收证据一律来自真实来源；搜不到就如实标注，绝不填假值。

---

## 二、立项与边界

5. **MVP 第一闭环**（A）
   先做能跑通的最小闭环再扩张，不追求一次到位。

6. **只推一条技术路线**（A）
   技术选型只给一个主推荐路线，并**写明拒绝其他路线的理由**。资料不足先补资料不急着吐技术栈。

7. **显式"不做什么"**（A）
   立项必须单列「本期不做」清单，防范围蔓延。

8. **半路接管先只读体检**（A）
   接手已有技能/半成品，首次先只读审计（结构、真源、owner、漂移），确认无治理风险再进入主流程。

---

## 三、架构与分层（通用原则，不绑定特定项目）

9. **Domain / Connector / Infra 分层**（B）
   - Domain Skills = 产品界面 + 业务逻辑
   - Connectors = 外部服务封装
   - Infra = 配置 / 日志 / 错误 / 监控

10. **确定性逻辑不交给概率系统**（B）
    资金 / 配额 / 权限等状态相关判定，必须纯函数确定性实现；概率模型只做路由 + 参数提取。

11. **保留逃生舱**（B）
    在对话/技能层之上保留确定通道（如 HTTP API）。对话层挂了仍有可用路径。

12. **可观测性从第一天起**（B）
    日志 / 监控 / 错误追踪在 MVP 阶段就要有最小骨架，不要等故障爆发才补。

---

## 四、改动门禁（轻量评估 vs 强制治理）

13. **窄修窄验**（A+B）
    单 owner 的常规改动（局部 UI、单文件、单测失败）直接完成，不套完整治理流程。

14. **自动升级为强制治理**（A+B，AI 自主判断）
    碰到以下任一类，AI **必须**先停手做评审文档 + 等签批，不得窄修：
    - 架构 / 数据模型 / 权限 / 支付 / 部署
    - 实施真源缺失或冲突
    - 跨 owner 改动
    - 用户说"停止"或"先别动"

15. **前端需求绝不扩大后端**（B）
    用户需求是看板 / 前端 / UI 类 → 只动前端，不顺手改后端逻辑。

---

## 五、实施真源与验收

16. **每阶段先写真源 dev-docs**（A）
    开发前写清目标 / owner / 验证方法 / 风险，再按子阶段开发。模板结合本项目路径改，不原样复制。

17. **验收要有硬证据**（A）
    验收 = 点击路径 + 截图 / 接口返回 / 测试结果 + 通过判断。**禁止"看起来成功了"式假成功**。

18. **防跑偏 guardrail**（A）
    定期查：真源是否落地、owner 是否清晰、验收证据是否齐、有无漂移。

---

## 六、Agent 宪法模板（顶层约束）

19. **项目类 / agent 类 / workflow 类技能必须 ships 一份宪法**（A，补通用缺口）
    Forge 这类技能时，让它产物自带 `AGENTS.md` / `CONSTITUTION.md`，固化：项目目标、技术红线、不可做之事、签批规则、安全红线。后续该技能每次运行先读它。
    - 纯 utility 技能（单文件即全部）可豁免，但要在 SKILL.md 写明 NON-mandate。

20. **宪法不可被会话临时覆盖**（A+B）
    会话里用户临时指令若与宪法冲突，AI 须提示冲突并请求确认，不得默默绕过。

---

## 七、安全三层闭环

21. **只读优先审计**（A）
    全面审计时先给风险 + 证据 + 修复顺序，不擅自改。

22. **写时自动判安全影响**（A）
    写代码 / 改配置 / 动数据 / 接第三方 / 发布时，自动判断本次安全影响（密钥、接口、权限、费用、公开访问）。

23. **长期红线进宪法**（A）
    安全红线沉淀进第六章的 Agent 宪法，约束后续所有会话。

24. **密钥与内部资料保护**（A+B）
    密钥 / 内部资料 / 用户数据永不进代码库、不进日志明文、不上行原文。

---

## 八、Git 与回滚保护

25. **Git 操作先风险判断**（A）
    初始化 / .gitignore / 远程 / 推送审批 / 回滚 / reset / revert，先做风险判断。

26. **混合 commit 不可整撤**（A）
    一个 commit 混多功能时，禁止直接整 commit revert；先判断要撤哪个功能，按文件 / hunk 拆。

27. **一键可回滚**（B）
    任何生产改动必须能一键回滚；无法回滚的视为高风险，需用户明确授权。

---

## 九、交接与防漂移

28. **上下文过长生成交接文本**（A）
    上下文超限或换 AI 时，生成含：现状 / 变更 / 验证证据 / 风险 / 下一步命令 的交接文本，不中断推进。

29. **重大决策入日志**（B+C）
    每次实质改动后向项目 memory / dev-docs 追加简短记录（做了什么、为什么、trade-off），跨会话可续。

---

## 十、上线前检查清单（MVP 阶段最小化但不可省）

30. **发布前必查**：
    - [ ] 环境变量 / 密钥就位且不泄露
    - [ ] 迁移脚本可逆
    - [ ] 回滚方案已验证
    - [ ] 监控 / 告警接通
    - [ ] 成本预估（按流量计费场景）
    - [ ] 公开访问风险（鉴权 / 限流 / 防刷）
    - [ ] 真源 / 验收证据齐

---

## 十一、持续进化（让技能越用越牛逼）

31. **非侵入反馈环**（C）
    默认只观察用户自然行为（采纳/纠正/重问/放弃），几乎不发问；弹窗问卷 / 每条评价是反面模式。详见 `feedback-loop.md`。

32. **拟真测试驱动验收**（C）
    上线前从真实社区 / 用户池捞真实问题跑验收，不靠自构测试自我迎合。详见 `simulation-testing.md`。

33. **度量尺 + 自审**（C）
    "最牛逼"必须可量化（rubric 打分），不可宣称；元技能要能递归自审。详见 `skill-review-rubric.md`。
